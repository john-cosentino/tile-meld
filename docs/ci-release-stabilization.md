# Release CI Stabilization Checkpoint

> Completion summary for the `fix/release-ci-stabilization` checkpoint,
> built on top of `712d031` (Merge identity, room lifecycle, dashboard, and
> tabletop updates). This is not a product feature phase — no application
> behavior was changed.

## Goal

Fix two release-blocking CI problems without weakening security scanning or
E2E coverage:

1. `pnpm audit --audit-level=high` failing on two vulnerable transitive
   `fast-uri` versions.
2. The single serial Playwright job exceeding GitHub Actions' one-hour job
   limit.

## Files changed

- `pnpm-workspace.yaml` — added a root `overrides` key.
- `pnpm-lock.yaml` — regenerated by `pnpm install` to resolve `fast-uri` to
  the patched versions under the new overrides.
- `.github/workflows/ci.yml` — replaced the single serial `e2e` job with a
  5-way parallel matrix, one job per Playwright project.
- `docs/ci-release-stabilization.md` (this file).

No other file changed. No `package.json` in any workspace was touched, no
application source, no `packages/engine`, no `packages/bot`, no database
migration, no Dockerfile, no Render config, and no product documentation
unrelated to this checkpoint.

## Part 1 — `fast-uri` transitive vulnerability

### Original vulnerable dependency paths

Two independent `fast-uri` versions were resolved into the graph, both
transitively via Fastify's Ajv-based schema tooling — never a direct
dependency of any workspace package:

```
fast-uri@3.1.3
├─┬ @fastify/ajv-compiler@4.0.5
│ └─┬ fastify@5.10.0
│   ├── @tile-meld/server (dependencies)
│   └─┬ fastify-type-provider-zod@4.0.2
│     └── @tile-meld/server (dependencies)
└─┬ ajv@8.20.0
  ├── @fastify/ajv-compiler@4.0.5 [deduped]
  ├─┬ ajv-formats@3.0.1
  │ └─┬ fast-json-stringify@7.0.1
  │   └── ... fastify@5.10.0 [deduped]
  └── fast-json-stringify@7.0.1 [deduped]

fast-uri@4.1.0
└─┬ fast-json-stringify@7.0.1
  ├─┬ @fastify/fast-json-stringify-compiler@5.1.0
  │ └─┬ fastify@5.10.0
  │   ├── @tile-meld/server (dependencies)
  │   └─┬ fastify-type-provider-zod@4.0.2
  │     └── @tile-meld/server (dependencies)
  └── fastify@5.10.0 [deduped]
```

### Exact overrides added

`pnpm-workspace.yaml` had no existing `overrides` key, so this was a clean
addition — no merge or duplicate top-level key:

```yaml
overrides:
  "fast-uri@>=3.0.0 <4.0.0": "3.1.4"
  "fast-uri@>=4.0.0 <5.0.0": "4.1.1"
```

Each override targets only its own major-version range, so the graph keeps
both required majors (3.x for the `ajv`/`ajv-formats` chain, 4.x for
`fast-json-stringify`) instead of forcing the whole graph onto one
potentially-incompatible major.

### Resulting `fast-uri` versions

`pnpm why fast-uri -r` after `pnpm install`:

```
fast-uri@3.1.4
├─┬ @fastify/ajv-compiler@4.0.5
│ └─┬ fastify@5.10.0 ...
└─┬ ajv@8.20.0 ...

fast-uri@4.1.1
└─┬ fast-json-stringify@7.0.1 ...

Found 2 versions of fast-uri
```

Exactly the two patched versions required (`3.1.4`, `4.1.1`), nothing else.
`pnpm install` reported `Packages: +2 -2` — the old `3.1.3`/`4.1.0` swapped
for the patched versions, with zero other packages touched. Fastify, Ajv,
and every other package resolve to the same versions as before.

### Audit result

```
$ pnpm audit --audit-level=high
No known vulnerabilities found
```

Exit code 0. The audit severity threshold (`--audit-level=high`) was not
changed, and the advisory was fixed, not suppressed or ignored (no
`.pnpmfile.cjs` allowlist, no `--ignore`/audit-exclusion mechanism was
added).

`fast-uri` was not added as a direct dependency anywhere — confirmed by
grepping every workspace `package.json` for the string, which returns no
matches.

## Part 2 — Playwright CI matrix split

### Original Playwright CI structure

One `e2e` job, `runs-on: ubuntu-latest`, `timeout-minutes: 60`, that
installed all three browser engines (`chromium firefox webkit`) and then
ran `npx playwright test` with no `--project` filter — all five projects
(`chromium`, `firefox`, `webkit`, `mobile-chrome`, `mobile-webkit`)
executing serially in that single job, gated further by
`playwright.config.ts`'s global `workers: 1`. A local full run took ~28
minutes; in CI this occasionally exceeded the 60-minute job limit.

### New matrix structure

`e2e` is now one job definition expanded by `strategy.matrix.include` into
5 parallel jobs, each scoped to exactly one Playwright project:

```yaml
strategy:
  fail-fast: false
  matrix:
    include:
      - project: chromium
        browser: chromium
      - project: firefox
        browser: firefox
      - project: webkit
        browser: webkit
      - project: mobile-chrome
        browser: chromium
      - project: mobile-webkit
        browser: webkit
```

- Job display name: `End-to-end (${{ matrix.project }})` — e.g.
  `End-to-end (mobile-webkit)`.
- `fail-fast: false` — one project's failure doesn't cancel the other four
  in-flight runners.
- `timeout-minutes: 30` — the matrix split (not this number) is the real
  fix for the original one-hour serial-job timeout; this value just needs
  to be a reasonable, non-masking ceiling for one project's own runtime.
  An initial estimate of 20 minutes turned out to be too tight: an
  isolated Mobile WebKit run has already taken slightly more than 20
  minutes when CI retries and real rate-limit backoff waits (both
  intentionally preserved, not something this checkpoint should shrink)
  landed in the same run. 30 minutes gives reasonable GitHub-runner
  headroom for that combination without returning to a single combined
  job, without touching production rate limits, and without being large
  enough to quietly mask a genuinely runaway job.
- PostgreSQL service block, health checks, `DATABASE_URL` env var, and the
  `pnpm --filter @tile-meld/server run migrate` step before tests are all
  byte-for-byte unchanged, just now present once per matrix job (each job
  gets its own isolated Postgres service container).
- Playwright CI retries are unchanged — `retries: process.env.CI ? 2 : 0`
  in `e2e/playwright.config.ts` was not touched.
- One worker per project is retained: `playwright.config.ts`'s global
  `workers: 1` still applies inside each matrix job's single-project run,
  preserving the existing per-IP-rate-limit-avoidance behavior documented
  there. Splitting into a matrix parallelizes *across* projects (each on
  its own runner/IP), not *within* a project's own test files.
- No test project was removed, no project's suite was sharded further, no
  production API rate limit was changed, no test was disabled, and no
  sleep was added anywhere.

### Project-to-browser mapping

| `matrix.project` | `matrix.browser` installed | Why |
| --- | --- | --- |
| `chromium` | `chromium` | Desktop Chrome engine |
| `firefox` | `firefox` | Desktop Firefox engine |
| `webkit` | `webkit` | Desktop Safari-proxy engine |
| `mobile-chrome` | `chromium` | Pixel 7 device emulation, same Chromium engine |
| `mobile-webkit` | `webkit` | iPhone 14 device emulation, same WebKit engine |

Each job installs only its required engine via
`npx playwright install --with-deps ${{ matrix.browser }}`, so a
`chromium`-engine job never pulls WebKit's or Firefox's OS-level
dependencies and vice versa.

Each job then runs `npx playwright test --project=${{ matrix.project }}`.

### Report/artifact behavior

`e2e/playwright.config.ts` sets `reporter: "html"`, which by default
writes to `<testDir-parent>/playwright-report/` — i.e. `e2e/playwright-report/`
relative to the `working-directory: e2e` steps use. That already matched
the pre-existing upload `path:`, so no path was fabricated or changed —
only the artifact `name` changed, from a single shared `playwright-report`
to `playwright-report-${{ matrix.project }}`. This is necessary, not
cosmetic: `actions/upload-artifact@v4` rejects a second upload of the same
artifact name within one workflow run, which the old shared name would hit
as soon as a second matrix job tried to upload — this is the structural
cause of a prior "no report files existed" warning once the job was
naively parallelized without renaming the artifact. `if: always()` is
retained so a failing project still uploads its diagnostic HTML report.

## CI security job

`security` job unchanged: still `needs: checks` (same dependency as
`e2e` — neither blocks or reorders around the other), still runs
`pnpm audit --audit-level=high`, `docker build`, and the Trivy scan with
`severity: "CRITICAL,HIGH"` and `ignore-unfixed: true`. Because `security`
and `e2e` are sibling jobs with the same single dependency (`checks`), the
audit step's pass/fail is never hidden behind or delayed by the E2E
matrix's runtime.

### Web test-count reconciliation (157 vs. the Phase 8 doc's "176")

`docs/phase-08-tabletop-layout.md`'s own validation table claims
`web 176 (↑ from 157)` — i.e. 157 tests before that phase's 19 new
`TabletopLayout.test.tsx` cases, 176 after. A fresh `pnpm run test` on this
branch (which changes no test or application source) reports **157**, not
176, so this was checked rather than assumed correct.

Verification: summing the individual `it`/`test` case counts across every
file in `apps/web/test/` (`grep -cE "^\s*(it|test)\(" <file>` per file,
by hand) totals exactly **157**, matching Vitest's own reported total
exactly — including all 19 cases in `TabletopLayout.test.tsx`, confirmed
present in that sum. `git log` shows `TabletopLayout.test.tsx` was added
whole, once, in commit `4b9897e` ("refactor(web): improve tabletop
information hierarchy") and has not been touched since; that same commit
also introduced `docs/phase-08-tabletop-layout.md` with the "176" claim
already in it.

Conclusion: this is a **pre-existing documentation arithmetic error in
`docs/phase-08-tabletop-layout.md`**, not a regression introduced by this
checkpoint and not a case of tests silently failing to be discovered. The
real pre-Phase-8 baseline was 138 web tests, not 157 as that document
stated — 138 + 19 new = 157, the number both this document and the fresh
run agree on today. No test was deleted, disabled, or lost; `docs/ci-
release-stabilization.md`'s "157" figure (present here since this
document's first draft) was already correct and required no numeric
change, only this reconciliation note. `docs/phase-08-tabletop-layout.md`
itself was not edited — correcting a different checkpoint's already-merged
documentation is outside this narrowly scoped checkpoint's remit.

## Known limitations

- `security` and `e2e` are independent sibling jobs (both `needs: checks`
  only) — this preserves the audit's visibility but means the audit does
  not literally block E2E from starting if the audit were to fail; it was
  already structured this way before this checkpoint, and reordering it
  was explicitly out of scope ("do not reorder the job in a way that hides
  audit failures" — the current arrangement doesn't hide it, it runs it in
  parallel with equal visibility in the PR checks list, which is the same
  visibility model as before).
- `timeout-minutes: 30` per matrix job is a reasoned estimate, corrected
  once already: an initial 20-minute value was raised to 30 after an
  isolated Mobile WebKit run took slightly over 20 minutes under CI
  retries and real rate-limit backoff waits. It has still not been
  independently measured against real GitHub Actions runner hardware in
  this checkpoint — the first few real CI runs on this branch remain the
  actual confirmation of comfortable headroom.
- The full five-project matrix was not re-run locally in this checkpoint
  (by design — the task called for one Chromium and one Mobile WebKit spec
  as a targeted, directly-relevant check, not a full re-run). The
  underlying `.spec.ts` files and `playwright.config.ts` were not modified
  at all this checkpoint, so the broader regression coverage already
  established in the prior tabletop-layout checkpoint continues to apply
  unchanged.
- `pnpm` itself reported a newer version is available (`11.13.0` →
  `11.16.0`) during `pnpm install`. Not upgraded — out of scope
  ("dependencies other than the narrow transitive override and lockfile
  resolution" were excluded), and the `packageManager` field pin was left
  untouched.
